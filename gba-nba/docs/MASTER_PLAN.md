# GBA AI Sales Cockpit — Next Best Action (NBA) Engine — Master Plan

> Status: **living plan (greenfield, designed from scratch)**. Service 3 of the GBA AI initiative.
> A sales manager logs in and sees a prioritized, AI-generated TASK QUEUE per client:
> who to call, what to (re)sell, why — with status (DONE etc.) and notes that PERSIST.
> Stateful (unlike the stateless reco/procure services) → MongoDB.

---

## 1. Vision

For each sales manager: an inbox of **actionable tasks**, each = *client × action × reason × priority*,
generated by AI from the signals we already compute, and **managed** through a lifecycle:

```
Manager opens cockpit →  prioritized task queue (their clients only)
  🔴 ТОВ "Європартсавто" — overdue 12d, debt 45k₴
      → Call: remind payment + reorder-due "brake pads" (~30d cycle, 34d elapsed)   [Done][Snooze][Note]
  🟡 ФОП Іванов — churning (−60% volume / 2mo)
      → Retain: offer X, Y (bought by similar clients, he lacks)                     [Done][Snooze][Note]
  🟢 ТОВ "Аккорд" — healthy, planned-buy window
      → Cross-sell: 3 new SKUs score 0.8+                                            [Done][Snooze][Note]
```

Each task is **persistent**: manager marks Done / Snooze / Dismiss, adds notes, sees history.
Feedback (done / sold / dismissed) flows back to improve scoring.

## 2. Where it sits (architecture)

```
GBA Console (manager UI: task inbox)
   │  HTTP (session/CSRF)
gba-server (.NET) ──► gba-nba (FastAPI, NEW)  ──► MongoDB   (task state: status, notes, history)
   orchestrator         NBA task engine        ──► gba-reco   (cross-sell candidates)  [HTTP]
                                               ──► gba-procure (reorder urgency)        [HTTP]
                                               ──► ConcordDb_V5 (read-only: signals)
```

- **gba-nba** is the 3rd Python service. It is the ONLY stateful one → owns MongoDB.
- It consumes reco + procure (already built) and queries ConcordDb_V5 read-only for signals.
- gba-server exposes it to the console (new controller, same pattern as Reco/Procurement).

## 3. Task lifecycle (the stateful core — your ask)

```
GENERATED ──(manager sees)──► OPEN ──► IN_PROGRESS ──► DONE
                                │           │
                                ├──► SNOOZED (until date) ──► OPEN
                                └──► DISMISSED (with reason)
            (regeneration: stale OPEN tasks expire; DONE/DISMISSED are immutable history)
```

- **DONE**: manager acted (called/sold). Optionally captures outcome (sold? amount?).
- **SNOOZED**: hidden until `snooze_until`, then reappears.
- **DISMISSED**: not relevant — captures a reason → strong negative feedback signal.
- **Idempotent regeneration**: the engine re-runs (nightly + on-demand). It must NOT duplicate a
  task the manager already has open/done. Dedup key = (manager_id, client_id, task_type, dedup_window).

## 4. Persistence model (MongoDB)

**Why Mongo, not ConcordDb_V5:** (a) ConcordDb is treated READ-ONLY (frozen source of truth);
mutable AI state must not live there. (b) Tasks are polymorphic documents — payload differs by
task_type (debt vs cross-sell vs reorder) — a natural fit for documents over rigid columns.
(c) Embedded notes + status_history are sub-documents read together with the task.
(d) Primary query is "manager's open tasks sorted by priority" — one indexed collection scan.

### Collection: `tasks`
```jsonc
{
  "_id": ObjectId,
  "task_key": "mgr:1042|client:410190|type:reorder_due|win:2026-06",  // dedup idempotency key (unique)
  "manager_id": 1042,
  "client_id": 410190,
  "client_name": "ТОВ Європартсавто",
  "task_type": "reorder_due",        // reorder_due | debt_followup | cross_sell | churn_winback | new_client_activation
  "title": "Подзвонити: час дозамовити",
  "reason": "Купують 'гальмівні колодки' кожні ~30д; минуло 34д",
  "priority": 87.4,                  // computed score (urgency × value × confidence)
  "urgency": "high",                 // critical | high | normal | low
  "status": "open",                  // generated|open|in_progress|done|snoozed|dismissed
  "payload": {                       // polymorphic, per task_type
    "products": [{"product_id": 25357460, "name": "...", "score": 0.81, "source": "reorder",
                  "expected_cycle_days": 30, "days_elapsed": 34}],
    "debt": {"overdue_days": 12, "amount": 45000, "currency": "UAH"}
  },
  // EXPLAINABILITY (v1): structured "why this task" — signals + human-readable factors.
  "signals": {"recency_days": 34, "frequency": 12, "monetary": 230000},
  "explanation": {
    "factors": ["34d since last buy vs ~30d cycle", "overdue 12d > 0d deferral", "RFM-M top 15%"],
    "source_signal": "reorder_due+debt", "confidence": 0.78
  },
  // CONTACT (v1): client contact embedded so the action is one-click (Client.MobileNumber/EmailAddress/Viber).
  "contact": {"phone": "+380...", "email": "...", "viber": "...", "preferred": "phone"},
  // SLA (v1): the task itself has a deadline + escalation to the head of sales.
  "due_date": ISODate,
  "sla_breached": false,
  "escalated_to": null,              // HeadSalesAnalytic User.ID if breached/escalated
  "notes": [
     {"author_id": 1042, "text": "Клієнт просив передзвонити в пʼятницю", "created_at": ISODate}
  ],
  "status_history": [
     {"from": "generated", "to": "open", "at": ISODate, "by": "system"},
     {"from": "open", "to": "done", "at": ISODate, "by": 1042, "outcome": {"sold": true, "amount": 12000}}
  ],
  "snooze_until": ISODate | null,
  "outcome": {"sold": true, "amount": 12000} | null,   // captured on DONE
  "ab_variant": "als",              // which generator/algorithm produced this (A/B attribution)
  "model_version": "nba-v1",
  "generated_at": ISODate,
  "updated_at": ISODate,
  "expires_at": ISODate              // OPEN tasks auto-expire if not acted on (regeneration hygiene)
}
```
**Indexes:** `{task_key:1} unique`; `{manager_id:1, status:1, priority:-1}` (the inbox query);
`{client_id:1, task_type:1}`; `{status:1, expires_at:1}` (expiry sweep); `{snooze_until:1}` (wake sweep);
`{status:1, due_date:1}` (SLA-breach sweep); `{escalated_to:1, status:1}` (head-of-sales view).

### Collection: `task_events` (append-only audit / feedback log)
Every transition + note as an immutable event → trains the scorer + analytics. (Could also be the
status_history source of truth; keep both for fast read (embedded) + full audit (events).)

### Collection: `manager_prefs` (v1 — needed for anti-spam)
Per-manager: muted task_types, muted (client_id, task_type) pairs with `mute_until` (set on DISMISS),
working hours (snooze defaults), max active tasks. Drives the throttling rules in §11.

## 5. Task types — ALL signals VERIFIED computable on ConcordDb_V5 (grounding workflow)

| task_type | trigger signal | verified data source | action |
|---|---|---|---|
| `reorder_due` | product past its purchase cycle | `[Order]` 6,267 + `[OrderItem]` 33,264, group by `Order.ClientAgreementID`, intervals from `Order.Created` | "time to reorder X" |
| `debt_followup` | overdue beyond deferral | `[Debt]` 5,727 + `[ClientInDebt]` 5,727 + `[Agreement]` (`NumberDaysDebt`, `AmountDebt`, `Debt.Days`, `Debt.Total`) | "remind payment" |
| `cross_sell` | high-score products client lacks | **gba-reco** /recommend (copurchase/ALS) — already built | "offer X,Y,Z" |
| `churn_winback` | order freq/volume drop | `[Order].Created` monthly trend per Agreement | "re-engage" |
| `new_client_activation` | new client, few orders | `Client.Created` + order count (1,801 new products/30d also enables new-SKU pushes) | "onboard / first cross-sell" |

⚠️ **Data caveats found (validate on real data before trusting):**
- Churn signal: dev DB shows a sharp spike (2026-05=2809, 2026-04=2399 orders) vs sparse 2025 (max ~295) → likely data-load artifact; churn baselines need real production history.
- `ProductGroupDiscount` (449,560 rows, has `ClientAgreementID`+`IsActive`) enables promo-targeted tasks — bonus signal for v2.

## 6. Scoring & prioritization

`priority = w_u·urgency + w_v·value + w_c·confidence` — config-driven weights, benchmark/feedback-tuned.
- **urgency**: overdue days, days-past-cycle, churn severity.
- **value**: client monetary (RFM-M), expected order size, debt amount.
- **confidence**: model score (reco) / signal strength.
- Diversity: cap tasks per client per day so the inbox isn't one client.
- Feedback loop: DISMISSED with reason and DONE-not-sold lower future priority of that type/client.

## 7. gba-nba service shape (mirrors hardened reco/procure patterns)

- Python 3.12, FastAPI, Pydantic v2, **pymongo/motor** for Mongo, httpx for reco/procure, SQLAlchemy read-only for signals.
- `app/{core/{config,logging,metrics}, data/{mongo, signals_repository, db}, clients/{reco_client, procure_client}, domain/models, services/{generators/*, scoring, lifecycle}, api/main}`.
- **Generators** (one per task_type) each emit candidate tasks → **scoring** → **lifecycle** (dedup/persist to Mongo).
- Endpoints:
  - `GET  /tasks/manager/{id}` — prioritized open queue (the inbox).
  - `POST /tasks/{id}/status` — done/snooze/dismiss (+outcome).
  - `POST /tasks/{id}/notes` — add note.
  - `POST /generate/manager/{id}` — regenerate for a manager (and a worker for all).
  - `GET  /health`, `/metrics`.
- Secrets via env; Mongo + DB read-only; Docker; tests; CI — same bar as the other two services.

## 11. Enhancements (user-approved)

### v1 must-haves (build into the core, not bolted on later)
- **Explainability** — every task carries `explanation.factors` (human-readable) + `signals` + `confidence`;
  UI shows a "Why this task?" expander. No black box → manager trust → adoption.
- **Contact-in-task** — embed `contact{phone,email,viber,preferred}` from `Client.MobileNumber/EmailAddress/Viber`
  so the action is one-click. Pulled at generation time.
- **Anti-spam / throttling** — `manager_prefs` + rules: max active tasks per manager; max tasks per client per day;
  on DISMISS set `(client_id, task_type) mute_until`; respect working hours; dedup by `task_key` (already core).
- **SLA + escalation** — `due_date` per task; a sweep flags `sla_breached`; breached high-urgency tasks
  `escalate_to` the client's/manager's `HeadSalesAnalytic` (role exists: UserRole.UserRoleType=6).

### Roadmap (later phases, user-approved)
- **Head-of-Sales dashboard + ROI** — team overview: tasks closed, conversion rate, revenue attributed to NBA
  (from `outcome.sold/amount`), debt exposure, per-manager load. Sells the system to leadership.
- **LLM action drafts** — reuse the AI-BI Ollama: generate a Ukrainian call-script / message draft per task,
  tailored to the client (turns "offer X" into ready-to-send copy). One-click actionable.
- **Feedback → ALL services** — `dismissed` / `done-not-sold` are ground-truth negatives fed back not just to
  NBA scoring but to gba-reco/gba-procure (rejected rec = negative example). Closes the whole system's loop.
- **A/B at task level** — `ab_variant` tags which generator/algorithm (v3.2 / copurchase / ALS) produced a task;
  measure which produces tasks that actually CONVERT (real outcomes), not just offline metrics. The true test.
